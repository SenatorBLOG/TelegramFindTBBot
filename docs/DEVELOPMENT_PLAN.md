4. Проверка: `curl -X POST https://telegramfindtbbot.fly.dev/webhook -d '{}'` → 401.

## P0 — Дыры в безопасности ✅ ГОТОВО (см. блок «Прогресс»)

### P0-1. Webhook без secret_token — можно подделать любой апдейт ⚠️ КРИТИЧНО

**Файлы:** `main.py` (`_run_webhook`), `config.py`

Эндпоинт `https://telegramfindtbbot.fly.dev/webhook` публичный, и бот принимает
любой POST. Любой, кто знает URL (а он угадываемый), может отправить фейковый
апдейт **от имени админа** (`from.id = ADMIN_USER_ID`): выполнить `/release`,
`/import`, апрувнуть спам-профиль через `mod_approve:` callback. Это и есть
вероятный канал, через который «дорасы просовывают спам» мимо модерации.

**Фикс:**
1. В `config.py` добавить `webhook_secret: str` — генерится один раз,
   хранится в Fly secrets (`WEBHOOK_SECRET`, например `secrets.token_urlsafe(32)`).
2. В `_run_webhook`:
   ```python
   await bot.set_webhook(url=url, secret_token=cfg.webhook_secret, drop_pending_updates=True)
   SimpleRequestHandler(dispatcher=dp, bot=bot, secret_token=cfg.webhook_secret).register(...)
   ```
   aiogram сам вернёт 401 на запросы без заголовка `X-Telegram-Bot-Api-Secret-Token`.
3. `flyctl secrets set WEBHOOK_SECRET="..."`.

**Приёмка:** `curl -X POST https://.../webhook -d '{}'` → 401. Бот работает как раньше.

### P0-2. HTML-инъекция в профилях — ломает публикацию и позволяет спуфинг ссылок

**Файлы:** `utils/formatters.py`, `handlers/admin.py` (preview), `handlers/profile.py`

Бот работает в `parse_mode=HTML` глобально (`bot.py`). Всё, что вводит юзер
(имя, био, язык, направление, даты), вставляется в HTML **без экранирования**
(`format_profile`). Последствия:
- Имя `<b`, био с `<3` или `A&B` → Telegram API «can't parse entities» →
  публикация профиля **падает** (вот почему у части людей «не получилось»).
- Злоумышленник вставляет `<a href="https://scam.example">Написать мне</a>`
  в био → фишинговая ссылка в карточке от имени бота.

**Фикс:** в `format_profile()` и везде, где юзерский текст попадает в сообщение
(превью модерации в `admin.py`, `/myprofile`), оборачивать поля в `html.escape()`:
```python
from html import escape
f"<b>✈️ {escape(profile.name)}</b>"
```
Поле `contact` — особый случай: там бот сам генерит `<a href="tg://user?id=...">`,
его не экранировать, но экранировать `user.full_name` внутри него
(`handlers/profile.py::_auto_contact`).

**Приёмка:** профиль с именем `<script>&"test"` создаётся, публикуется и
отображается литерально. Юнит-тест на `format_profile` с спецсимволами.

### P0-3. Спам-фильтр — решето

**Файлы:** `utils/word_filter.py`, `main.py` (middlewares, `allowed_updates`)

Текущий фильтр: 11 английских слов. Обходится за секунду. Конкретные дыры:

1. **`edited_message` вообще не обрабатывается.** Классический обход: постишь
   «привет», через минуту редактируешь в спам. Бот эти апдейты даже не
   получает (`dp.resolve_used_update_types()` отдаёт только `message` +
   `callback_query`). Фикс: зарегистрировать `dp.edited_message.outer_middleware(SpamMiddleware())`
   и хендлер-заглушку, чтобы тип апдейта попал в allowed_updates; прогонять
   отредактированный текст через тот же фильтр + удалять.
2. **Нет детекта ссылок** — главный вектор спама: `t.me/joinchat/...`,
   `telegram.me`, `wa.me`, любые URL от свежих аккаунтов. Фикс: в `is_spam()`
   добавить regex на приглашения (`t\.me/(joinchat|\+)`, `telegram\.(me|dog)`)
   и опцию «URL от не-участника-со-стажем = спам» (см. п.4).
3. **Нет русских слов и нет нормализации.** Добавить RU-словарь (казино,
   заработок, инвестиции, крипта, эскорт, интим…) и нормализацию перед
   проверкой: lower → замена look-alike кириллицы/латиницы (с→c, о→o, а→a…) →
   схлопывание повторов и пробелов («к а з и н о» → «казино»).
4. **Новички без ограничений.** Завести таблицу `chat_members(user_id, first_seen)`
   (заполнять в `TrackUserMiddleware`): если юзер в группе < 24ч или < 3
   сообщений — любое сообщение со ссылкой/форвардом/упоминанием удалять.
   Это отсекает 90% залётных спамеров без капчи.
5. **Ноль видимости у админа.** При удалении спама — форвардить копию админу
   в личку с кнопками `[Забанить] [Ложное срабатывание]`. `[Забанить]` →
   `bot.ban_chat_member`. Лог удалений в таблицу `spam_log`.
6. **Профили не проверяются фильтром.** При `MODERATION=false` спам можно
   опубликовать прямо в карточке. Фикс: в `_finalize()` прогонять name/bio
   через `is_spam()` + запрет URL в имени.

**Приёмка:** сценарии «edit в спам», «t.me-ссылка от новичка», «кириллический
спам с заменой букв» — сообщение удалено, админ получил уведомление.

### P0-4. Баг фейкового user_id в `/import`

**Файл:** `handlers/admin.py::cmd_import`

`fake_user_id = -int(time.time() * 1000) % (10 ** 12)` — в Python результат
`%` от отрицательного числа **положительный**. Задумка «отрицательный id,
чтобы не пересекаться с реальными» не сработала: получаем положительное число
до 10¹², которое теоретически может совпасть с настоящим Telegram id.

**Фикс:** `fake_user_id = -(int(time.time() * 1000) % 10**12)` (минус снаружи).
Проверить, что `profiles.user_id` и запросы нормально живут с отрицательными
значениями (живут — это просто INTEGER).

**Приёмка:** `/import Test | Spain | June 2027` → в БД `user_id < 0`.

### P0-5. Ротация токена бота + перевесить UptimeRobot

Токен бота засветился в логах, чатах и curl-командах; `.env` с ним лежит
локально. Плюс UptimeRobot до сих пор мониторит **мёртвый Render-URL**.

**Действия (руками, не код):**
1. @BotFather → `/revoke` → новый токен → `flyctl secrets set BOT_TOKEN=...`.
2. UptimeRobot: монитор на `https://telegramfindtbbot.fly.dev/health` (GET, 5 мин),
   старый монитор на onrender.com удалить.
3. Удалить реальный токен из локального `.env`-примера, если файл когда-нибудь
   пойдёт в паблик (сейчас в .gitignore — ок).

## P1 — Баги и корректность ✅ ГОТОВО (см. блок «Прогресс»)

### P1-1. `/search`, `/myprofile`, `/deleteprofile` работают в группе и мусорят

**Файлы:** `handlers/search.py`, `handlers/profile.py`

`/profile` в группе корректно редиректит в личку кнопкой, а `/search` — нет:
запускает FSM-визард **прямо в группе**, клавиатуры и шаги видны всем.
`/myprofile` и `/deleteprofile` тоже отвечают в группе (палят данные + мусор).

**Фикс:** общий guard в начале каждого хендлера — если `message.chat.type != "private"`,
ответить deep-link-кнопкой (`t.me/<bot>?start=search`) и выйти. В `common.py::cmd_start`
добавить обработку payload `search` по аналогии с `profile`. Бонус: удалять
команду юзера из группы (`bot.delete_message`), чтобы вообще чисто.

**Приёмка:** любая из команд в группе → одна кнопка-редирект, визард только в DM.

### P1-2. Статистика юзеров врёт

**Файлы:** `main.py` (порядок middleware), `repositories/user_repo.py`, `db.py`

`TrackUserMiddleware` апсертит **всех, кто написал в группе**, а не юзеров бота —
отсюда «5 юзеров, потом 1». Плюс он стоит **до** SpamMiddleware, так что спамеры
тоже попадают в таблицу.

**Фикс:**
1. Поменять порядок: `SpamMiddleware` регистрировать первым (outer), TrackUser — после.
2. В `users` добавить колонку `source` (`private`/`group`) или булево
   `interacted_privately`; `/stats` считать «Users» только по private.
3. В `/stats` добавить: юзеров за 7 дней, профилей за 7 дней, число импортов
   (`user_id < 0`).

**Приёмка:** `/stats` показывает раздельно: юзеры бота (DM) / писавшие в группе.

### P1-3. Пагинация поиска: кэш в памяти + доверие к user_id из callback

**Файл:** `handlers/search.py`

`_search_cache` — глобальный dict: (а) не чистится → утечка, (б) умирает при
рестарте → «Search session expired», (в) `sp:{user_id}:{offset}` берёт user_id
из callback-data, т.е. можно листать чужую сессию (не страшно, но грязно).

**Фикс:** хранить фильтры в FSM (`state.update_data`) вместо глобального dict'а,
в callback класть только offset, user_id брать из `cb.from_user.id`. Кэш удалить.

**Приёмка:** поиск переживает рестарт бота (пока жив fsm.db), чужой offset не работает.

### P1-4. Комментарий «update it» врёт: карточка пересоздаётся, а не редактируется

**Файл:** `services/topic_service.py::update_profile`

Метод называется update, а по факту delete + send (карточка прыгает вниз топика,
старые реплаи на неё сиротеют). Для текстовых карточек можно честно редактировать
(`edit_message_text` / `edit_message_caption`), fallback на delete+send только если
редактирование невозможно (сменился тип: было фото → стало без фото).

**Приёмка:** правка био без смены фото не двигает карточку в топике.

### P1-5. Мёртвый keepalive-код

**Файл:** `main.py::_keepalive`

На Fly с `auto_stop_machines='off'` самопинг бесполезен (он был нужен для
Render). Убрать таск и функцию, мониторинг снаружи делает UptimeRobot (P0-5).
Меньше кода — меньше сюрпризов.
